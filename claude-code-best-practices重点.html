Claude Code Best Practices

最佳实践完整指南 · 智能体式编码（agentic coding）
来源：Anthropic 官方文档 — code.claude.com/docs/en/best-practices
（原《Claude Code: Best practices for agentic coding》工程博客已迁移并扩充为此官方文档）
整理日期：2026-05-29

Claude Code 是一个智能体式（agentic）编码环境。它不像聊天机器人那样回答问题后等待——它能读取你的文件、运行命令、修改代码，并在你旁观、纠偏或离开时自主推进问题。

这改变了工作方式：你不再自己写代码让 Claude 审查，而是描述你想要什么，让 Claude 决定如何构建。Claude 负责探索、规划、实现。但这种自主性有学习曲线，Claude 在一定约束内工作，你需要理解这些约束。

核心约束：上下文窗口

几乎所有最佳实践都基于同一个约束：Claude 的上下文窗口填得很快，而且越满性能越差。

  • 上下文窗口装着你的整个对话：每条消息、Claude 读过的每个文件、每条命令的输出。
  • 一次调试或代码库探索可能就消耗几万 token。
  • LLM 性能随上下文填满而下降——快满时 Claude 会"忘记"早先的指令、犯更多错误。
  • 上下文窗口是最重要的待管理资源。可用自定义状态栏持续追踪用量。

1给 Claude 一个能自己验证工作的方法

给 Claude 一个它能运行的检查——测试、构建、截图对比。这是"需要你盯着的会话"和"你能走开的会话"之间的区别。

Claude 在"看起来完成了"时就停下。没有可运行的检查，"看起来完成"就是唯一信号，于是你变成了验证循环：每个错误都得等你发现。给它一个能产出"通过/失败"的东西，循环就能自我闭合——Claude 干活、跑检查、读结果、迭代直到通过。

检查可以是任何能在对话中读到信号的东西：测试套件、构建退出码、linter、把输出和 fixture 做 diff 的脚本、或浏览器截图与设计稿对比。

策略改进前改进后
提供验证标准"实现一个校验邮箱的函数""写一个 validateEmail 函数。示例用例：user@example.com→true，invalid→false，user@.com→false。实现后运行测试"
可视化验证 UI 改动"把仪表盘弄好看点""[贴截图] 实现这个设计。完成后截图并与原图对比，列出差异并修复"
针对根因而非症状"构建失败了""构建报错：[贴错误]。修复并验证构建成功。解决根因，不要把错误压下去"

检查存在后，决定它对"停止"的把关强度：

  • 单条 prompt 内：在同一条消息里要求 Claude 跑检查并迭代。
  • 跨会话：把检查设为 /goal 条件，独立评估器每轮重新检查直到成立。
  • 作为确定性闸门：用 Stop hook 把检查跑成脚本，未通过就阻止结束本轮（连续阻止 8 次后会被覆盖结束）。
  • 第二意见：用验证 subagent 或动态 workflow，让全新模型尝试反驳结果——干活的不当评分的。

让 Claude 展示证据而非声称成功：测试输出、运行的命令及返回、结果截图。审查证据比你自己重跑验证更快，对你没盯着的会话也有效。

2先探索，再规划，最后写代码

把"研究与规划"和"实现"分开，避免解决错误的问题。

让 Claude 直接跳到写代码，可能产出解决错问题的代码。用 plan mode 把探索和执行分开。推荐的四阶段工作流：

  1. 探索（Explore） — 进入 plan mode。Claude 读文件、回答问题，不做修改。
    read /src/auth and understand how we handle sessions and login...
  2. 规划（Plan） — 让 Claude 创建详细实现计划。
    I want to add Google OAuth. What files need to change? Create a plan.
    按 Ctrl+G 可在文本编辑器里直接编辑计划。
  3. 实现（Implement） — 切出 plan mode，让 Claude 写代码并对照计划验证。
    implement the OAuth flow from your plan. write tests... run the test suite and fix any failures.
  4. 提交（Commit） — 让 Claude 用描述性信息提交并开 PR。
    commit with a descriptive message and open a PR
何时跳过规划：plan mode 有用但有开销。范围清晰、改动小的任务（改 typo、加日志、重命名变量）直接做。规划在你不确定方案、改动跨多文件、或不熟悉被改代码时最有用。如果你能用一句话描述这个 diff，就跳过规划。

3在 prompt 中提供具体上下文

指令越精确，需要纠正的次数越少。

Claude 能推断意图，但读不了你的心。引用具体文件、说明约束、指向示例模式。

策略改进前改进后
限定范围"给 foo.py 加测试""为 foo.py 写测试，覆盖用户已登出的边界情况。不要用 mock。"
指向信息源"为什么 ExecutionFactory 的 api 这么怪？""查它的 git 历史，总结 api 是怎么演变的"
引用已有模式"加一个日历组件""看首页现有组件怎么实现，HotDogWidget.php 是好例子，按这个模式实现……只用已有的库"
描述症状"修登录 bug""会话超时后登录失败，检查 src/auth/ 尤其 token 刷新。先写复现的失败测试，再修"
模糊 prompt 在探索阶段也有用（"这个文件你会改进什么？"能发现你想不到的点），前提是你能承担纠偏成本。

提供丰富内容

  • 用 @ 引用文件，而非描述代码在哪。Claude 会先读再答。
  • 直接粘贴图片（复制粘贴或拖拽）。
  • 给 URL 指向文档/API 参考。用 /permissions 把常用域名加白名单。
  • 管道喂数据：cat error.log | claude。
  • 让 Claude 自己取：告诉它用 Bash、MCP 工具或读文件来拉取上下文。

4配置你的环境

写好 CLAUDE.md

运行 /init 基于当前项目结构生成初版 CLAUDE.md，再逐步打磨。

CLAUDE.md 是 Claude 在每次对话开始时都会读的特殊文件。包含 Bash 命令、代码风格、工作流规则——给 Claude 它无法从代码推断的持久上下文。格式无要求，但保持简短、人类可读：

# Code style
- Use ES modules (import/export) syntax, not CommonJS (require)
- Destructure imports when possible (eg. import { foo } from 'bar')

# Workflow
- Be sure to typecheck when you're done making a series of code changes
- Prefer running single tests, and not the whole test suite, for performance

CLAUDE.md 每次会话都加载，所以只放普遍适用的内容。仅有时相关的领域知识用 skills 代替。保持精简，每一行都问："删掉这行会让 Claude 犯错吗？" 不会就删。臃肿的 CLAUDE.md 会让 Claude 忽略你真正的指令！

✅ 应包含❌ 应排除
Claude 猜不到的 Bash 命令读代码就能搞清的东西
与默认不同的代码风格规则已知的标准语言惯例
测试说明与首选 test runner详细 API 文档（链接即可）
仓库礼仪（分支命名、PR 规范）频繁变动的信息
项目特有的架构决策冗长解释或教程
开发环境怪癖（必需环境变量）逐文件的代码库描述
常见坑或非显而易见的行为"写干净的代码"这类不言自明的实践
  • Claude 总做你不想要的事（尽管有规则）→ 多半文件太长、规则被淹没。
  • 把 CLAUDE.md 当代码对待：出问题时审查、定期修剪、观察行为是否真变了来测试改动。
  • 可加强调词（IMPORTANT、YOU MUST）提升遵守度。提交到 git 让团队贡献。

可用 @path/to/import 导入其他文件。可放置位置：家目录 ~/.claude/CLAUDE.md（所有会话）、项目根 ./CLAUDE.md（共享）、./CLAUDE.local.md（个人，加 .gitignore）、父目录（monorepo 自动拉入）、子目录（按需拉入）。

配置权限

用 auto mode 让分类器处理审批、/permissions 给特定命令加白名单、/sandbox 做 OS 级隔离。
  • Auto mode：独立分类器审查命令，只拦截有风险的（范围升级、未知基础设施、恶意内容驱动）。
  • 权限白名单：放行确知安全的工具，如 npm run lint、git commit。
  • 沙箱：OS 级隔离，限制文件系统和网络访问，让 Claude 在边界内更自由工作。

使用 CLI 工具

告诉 Claude 用 gh、aws、gcloud、sentry-cli 等与外部服务交互。

CLI 工具是与外部服务交互最省上下文的方式。用 GitHub 就装 gh。Claude 也擅长学不熟的工具：Use 'foo-cli --help' to learn it, then solve A, B, C.

连接 MCP 服务器

运行 claude mcp add 连接 Notion、Figma 或你的数据库。

可让 Claude 从 issue tracker 实现功能、查数据库、分析监控数据、集成 Figma 设计、自动化工作流。

设置 hooks

对"每次都必须发生、零例外"的操作用 hooks。

Hooks 在工作流特定节点自动运行脚本。不同于 CLAUDE.md（建议性的），hooks 是确定性的，保证操作发生。Claude 能帮你写："写一个在每次文件编辑后运行 eslint 的 hook"。运行 /hooks 浏览已配置的。

创建 skills

在 .claude/skills/ 创建 SKILL.md 给 Claude 领域知识和可复用工作流。
---
name: api-conventions
description: REST API design conventions for our services
---
# API Conventions
- Use kebab-case for URL paths
- Use camelCase for JSON properties
- Always include pagination for list endpoints
- Version APIs in the URL path (/v1/, /v2/)

对有副作用、想手动触发的工作流，用 disable-model-invocation: true，再用 /skill-name 调用。

创建自定义 subagents

在 .claude/agents/ 定义专门助手，Claude 可委派隔离任务。

Subagents 在自己的上下文里运行，有自己的允许工具集。适合读很多文件或需要专注、又不想弄乱主对话的任务。明确告诉 Claude："Use a subagent to review this code for security issues."

安装插件

运行 /plugin 浏览市场。插件无需配置即添加 skills、工具和集成。

插件把 skills、hooks、subagents、MCP 打包成单个可安装单元。用类型语言可装 code intelligence 插件，获得精确符号导航和编辑后自动错误检测。

5高效沟通

问代码库问题

像问资深工程师那样问 Claude。

上手新代码库时用 Claude 学习探索：日志怎么工作？怎么新建 API 端点？foo.rs 第 134 行的 async move 干嘛的？这是高效 onboarding 工作流，无需特殊 prompt，直接问。

让 Claude 来访谈你

对较大的功能，先让 Claude 用 AskUserQuestion 工具访谈你。
I want to build [brief description]. Interview me in detail using the
AskUserQuestion tool. Ask about technical implementation, UI/UX, edge cases,
concerns, and tradeoffs. Don't ask obvious questions, dig into the hard parts
I might not have considered. Keep interviewing until we've covered everything,
then write a complete spec to SPEC.md.

spec 完成后，开一个全新会话来执行——干净上下文专注实现。最有用的 spec 是自包含的：点名文件和接口、说明什么不在范围内、以端到端验证步骤收尾。把 spec 打磨精确，比花时间盯着实现更值。

6管理你的会话

对话是持久且可逆的，善用这点！

尽早、频繁地纠偏

一发现 Claude 跑偏就纠正。
  • Esc：中途停止，上下文保留，可重定向。
  • Esc + Esc 或 /rewind：恢复之前的对话和代码状态。
  • "Undo that"：让 Claude 回滚改动。
  • /clear：在不相关任务之间重置上下文。
同一问题纠正超过两次→上下文已被失败方案污染。/clear 重新开始。干净会话 + 更好 prompt，几乎总胜过长会话 + 累积纠正。

积极管理上下文

在不相关任务之间运行 /clear。
  • 任务间频繁 /clear 彻底重置。自动压缩会保留代码模式、文件状态、关键决策。
  • 更可控：/compact Focus on the API changes。
  • 只压缩部分：/rewind 选检查点，选 Summarize from here / up to here。
  • 在 CLAUDE.md 自定义压缩："When compacting, always preserve the full list of modified files and any test commands"。
  • 不需留存的小问题用 /btw，答案在可关闭浮层，不进对话历史。

用 subagents 做调查

用 "use subagents to investigate X" 委派研究。它们在独立上下文探索，保持主对话干净。

Claude 研究代码库时会读很多文件，全消耗你的上下文。Subagents 在独立上下文窗口运行并只回报摘要，是最强工具之一。也可用于实现后验证：use a subagent to review this code for edge cases。

用 checkpoints 回退

你发的每条 prompt 都会创建检查点。可恢复对话、代码或两者到任意之前的检查点。

双击 Escape 或 /rewind 打开菜单。可以让 Claude 尝试有风险的做法，不行就 rewind 换路子。检查点跨会话持久。

检查点只追踪 Claude 做的改动，不含外部进程。这不能替代 git。

恢复对话

用 /rename 给会话命名，像分支一样对待。

claude --continue 接最近会话，claude --resume 从列表选。给会话起描述性名字如 oauth-migration 方便日后查找。

7自动化与规模化

非交互模式（headless）

在 CI、pre-commit hook 或脚本里用 claude -p "prompt"。加 --output-format stream-json --verbose 输出流式 JSON。
# 一次性查询
claude -p "Explain what this project does"
# 给脚本用的结构化输出
claude -p "List all API endpoints" --output-format json
# 实时处理的流式输出
claude -p "Analyze this log file" --output-format stream-json --verbose

运行多个 Claude 会话

并行运行多个会话来加速开发、跑隔离实验或启动复杂工作流。
  • Worktrees：隔离的 git checkout 里跑独立 CLI 会话，编辑不冲突。
  • 桌面应用：可视化管理多个本地会话，各在自己的 worktree。
  • Claude Code on the web：在 Anthropic 托管云、隔离 VM 里跑。
  • Agent teams：自动协调多会话，共享任务、消息和 team lead。

Writer/Reviewer 模式：会话 A 实现功能 → 会话 B 在全新上下文审查（不偏袒刚写的代码）→ 会话 A 处理反馈。也可一个写测试、另一个写代码通过测试。

跨文件 fan-out

循环对每个任务调用 claude -p。用 --allowedTools 为批量操作限定权限。
for file in $(cat files.txt); do
  claude -p "Migrate $file from React to Vue. Return OK or FAIL." \
    --allowedTools "Edit,Bash(git commit *)"
done

先在 2-3 个文件测试打磨 prompt，再跑全集。也可接入管道：claude -p "..." --output-format json | your_command。

用 auto mode 自主运行

claude --permission-mode auto -p "fix all lint errors"

分类器在命令运行前审查，拦截风险操作，让常规工作免提示通过。-p 非交互运行若反复被拦会中止（无用户可回退）。

加一个对抗式审查步骤

把任务当作完成之前，让一个 subagent 在全新上下文审查 diff 并报告缺口。

全新 subagent 只看 diff 和你给的标准，看不到产生改动的推理，按自己的标准评估。正确性检查用内置 /code-review skill；对照计划检查则自己写 prompt：

Use a subagent to review the rate limiter diff against PLAN.md. Check that
every requirement is implemented, the listed edge cases have tests, and
nothing outside the task's scope changed. Report gaps, not style preferences.
被要求找缺口的审查者通常总会报告一些，即使工作没问题。追逐每个发现会导致过度工程化。告诉审查者只标记影响正确性或既定需求的缺口，其余视为可选。

8避免常见失败模式

失败模式修复
"厨房水槽"会话：一个任务里穿插无关的事，上下文塞满无关信息。不相关任务之间 /clear。
反复纠正：做错→纠正→还错→再纠正，上下文被失败方案污染。两次纠正失败后 /clear，写更好的初始 prompt。
过度指定的 CLAUDE.md：太长，重要规则被淹没。无情修剪；能做对的删掉或转成 hook。
信任却不验证：看似合理但不处理边界情况。永远提供验证（测试/脚本/截图），验证不了别上线。
无限探索：不限范围的"调查"读几百个文件填满上下文。窄范围限定，或用 subagents 让探索不消耗主上下文。

9培养你的直觉

这些模式不是铁律，而是普遍有效的起点，未必对每种情况最优。

  • 有时你应该让上下文累积——深陷一个复杂问题，历史很有价值。
  • 有时应该跳过规划，让 Claude 自己摸索——任务是探索性的。
  • 有时模糊 prompt 恰恰对——你想先看 Claude 如何解读，再去约束。

留意什么有效。Claude 产出很棒时注意你做了什么（prompt 结构、上下文、模式）；卡住时问为什么（上下文太杂？prompt 太模糊？任务太大？）。随时间你会培养出任何指南都无法捕捉的直觉。
