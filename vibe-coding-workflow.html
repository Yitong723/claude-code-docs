AI-Native Workflow

高效 Vibe Coding 流程
从 Idea → Demo → Product

核心思路一句话：让 Claude 写代码之前，先逼它（和你自己）把「要做什么、怎么验证」想清楚——Vibe Coding 的效率不在于打字快，而在于把规划、上下文、验证这三件事做扎实。

全局心法（贯穿始终）

原则含义
Explore → Plan → Code → Commit官方最推荐的基本节奏。绝大多数「AI 写出一堆垃圾」都是跳过了前两步。
上下文 = 生产力模型只能用它「看得到」的东西。你的工作很大程度上是上下文工程（CLAUDE.md / 截图 / 文档 / 引用文件）。
可验证性优先每一步都要有「怎么知道它对了」的手段：测试、运行截图、类型检查。没有验证的 vibe coding 会在 product 阶段崩盘。
小步提交每个能跑通的节点就 commit，方便回滚。AI 改坏了就 git reset，比让它「修复」更快。
1

Idea → 想清楚（Explore + Plan）

目标：在写任何代码前，把模糊的 idea 变成明确的需求和技术方案。

先用对话 / 头脑风暴展开想法

  • 用 /brainstorming（或直接对话）逼自己回答：用户是谁？核心 flow 是什么？最小可行切片是什么？MVP 砍到什么程度？
  • 让 Claude 扮演产品质疑者：「帮我挑战这个 idea 的 3 个最大风险」。

进入 Plan Mode（关键）

  • 用 Shift+Tab 切到 plan mode。Plan mode 下 Claude 只读不写，会先探索代码 / 调研技术栈，产出一份实施计划让你审。
  • 对新项目：让它先调研技术选型（「对比 Next.js vs Vite + Express 做这个 demo 的取舍」）。

沉淀技术方案为文件

  • 让 Claude 把计划写成 PLAN.md / SPEC.md（/writing-plans 技能就是干这个的）。好处：计划是文件 = 可审查、可版本化、可在新 session 里复用，不会随对话丢失。
💡 Tip：这一步最值钱。一份你亲自审过的 plan，能让后面 80% 的代码一次成型。不要急着看到代码。
2

搭好「上下文地基」（项目初始化）

目标：让 Claude 在整个项目周期里都「懂」你的代码和规范。

/init 生成 CLAUDE.md

  • 在项目根目录运行 /init，它会扫描代码库生成 CLAUDE.md（项目记忆）。
  • 手动补充：常用命令（build/test/lint/dev）、目录结构约定、代码风格、「不要做什么」的红线。这个文件每次都会自动加载进上下文。

分层记忆

  • ./CLAUDE.md（项目级，进 git，团队共享）
  • ~/.claude/CLAUDE.md（个人全局偏好）
  • 子目录可放局部 CLAUDE.md。

接入外部上下文（按需）

  • MCP servers：连数据库、Figma、Sentry、文档站等，让 Claude 能直接读真实数据 / 设计稿。
  • 截图 / 设计稿：做前端时直接把 UI 截图拖进来，让它「照着实现」。
3

Plan → Demo（快速原型）

目标：用最快速度做出能跑、能看、能点的原型，验证 idea 成不成立。

按计划分块实现

  • 让 Claude 按 PLAN.md 一个模块一个模块地实现，而不是「一次给我全部」。
  • 用 /executing-plans 或 /subagent-driven-development 这类技能，把计划拆成带 review checkpoint 的执行流。

demo 阶段允许「糙」，但要能跑

  • 优先打通 happy path（核心主流程跑通），先不管边界情况。
  • 立刻运行：用 /run 启动 app、/verify 确认行为符合预期。让 Claude 自己跑 dev server、截图、读报错——形成「写 → 跑 → 看 → 改」的闭环。

前端原型用 /frontend-design

  • 避免千篇一律的「AI 味」UI，产出有设计感的界面。

每个里程碑 commit

  • demo 能点通了就提交。这是你的安全网。
💡 Tip：demo 阶段最忌「完美主义」。目标是最快拿到反馈（给自己、给用户、给合伙人看），而不是代码质量。质量留给阶段四。
4

Demo → Product（工程化加固）

目标：把「能跑的 demo」变成「敢上线、能维护、不会半夜被叫起来」的产品。这一步是 vibe coding 最容易翻车、也最体现功力的地方。按重要性排序：

① 测试驱动（最重要的护城河）

  • 用 /test-driven-development：让 Claude 先写测试（基于真实预期，不是它脑补的），你确认测试合理，再让它写实现直到测试通过。
  • 这是防止「AI 改 A 弄坏 B」的根本手段。product 阶段没有测试 = 裸奔。

② 系统化调试，而不是「再试一次」

  • 遇到 bug 用 /systematic-debugging：先复现、定位根因、再改，而不是让 AI 乱猜补丁。
  • 善用 git：改坏了 git reset --hard 回到上一个 commit，比纠缠「修复」高效。

③ 代码审查

  • 自查：/code-review（按 effort 分级）、/simplify（清理冗余）。
  • 重要分支 / PR：/code-review ultra 启动多 agent 云端深度审查，或 /security-review 做安全审查。
  • 收到 review 意见时用 /receiving-code-review——不要盲目照单全收，要核实建议是否真的对。

④ 完成前强制验证

  • 用 /verification-before-completion：声称「做完了 / 修好了」之前，必须真的跑命令、看输出。杜绝「我觉得应该好了」。

⑤ 并行化提速（项目变大后）

  • Subagents：把独立任务（写测试 + 更新文档 + 重构模块）派给并行 agent，/dispatching-parallel-agents。
  • Git worktrees：/using-git-worktrees 让多个任务在隔离的工作区并行，互不污染。

⑥ 自动化与防护栏

  • Hooks：在 settings.json 配 hook，实现「每次保存自动 lint/format」「提交前自动跑测试」等——这些是确定性的，比每次提醒 Claude 可靠。用 /update-config 配置。
  • CI/CD：让 Claude 帮你写 GitHub Actions，把测试 / lint / 构建自动化。

⑦ 收尾

  • /finishing-a-development-branch：完成后规范地 merge / 开 PR / 清理分支。
5

贯穿全程的「提效杠杆」

这些不属于某一步，但用好了能整体翻倍：

  • 记忆系统：用 # 快捷记忆，或让它写入 CLAUDE.md / memory 文件，跨 session 记住你的偏好和项目约定。
  • 自定义 Slash Commands / Skills：把你反复做的流程封装成可复用技能（skill-creator）。这就是 AI-native 的精髓。
  • /clear 管理上下文：任务切换时清空上下文，避免旧任务干扰、避免 token 浪费。长任务自动压缩也会保留摘要。
  • 图片 / 文档输入：报错截图、设计稿、API 文档直接喂进去，远比文字描述高效。
  • Plan mode 当默认：养成「先 Shift+Tab 看计划再放行」的肌肉记忆。

给你的 3 个最高 ROI 建议

1

永远先 plan 再 code——这一个习惯就能消除大半返工。

2

product 阶段把测试当成不可谈判项——这是 demo 和真产品的分水岭。

3

把重复流程沉淀成 CLAUDE.md + 自定义 skill——把这套方法迁移到任何新产品上即可。
