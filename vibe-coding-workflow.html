AI-Native Workflow

高效 Vibe Coding 流程
从 Idea → Demo → Product

核心思路一句话：让 Claude 写代码之前，先逼它（和你自己）把「要做什么、怎么验证」想清楚——效率不在于打字快，而在于把规划、上下文、验证这三件事做扎实。

全局心法（贯穿始终）

原则含义来源
上下文窗口是根本约束模型只能用它「看得到」的东西，且上下文越满表现越差。你的工作很大程度上是上下文工程：该喂的喂（CLAUDE.md / 截图 / 引用文件），该清的清（/clear）。官方
Explore → Plan → Code → Commit官方最推荐的四段节奏。绝大多数「AI 写出一堆垃圾」都是跳过了前两步。官方
给 Claude 能自跑的验证手段官方原话：这是整篇唯一只学一条也要学的。给它测试 / 构建退出码 / 截图对比，它就能自己「做→跑→读结果→改」闭环，而不是把你变成验证循环。官方
小步提交 + 检查点每个能跑通的节点就 commit。改坏了用 git reset，或 /rewind（Esc+Esc）回到 Claude 改动前的检查点——比让它「修复」更快。官方
1

Idea → 想清楚（Explore + Plan）

目标：在写任何代码前，把模糊的 idea 变成明确的需求和技术方案。

① 让 Claude 反过来「面试你」官方

  • 官方推荐做大功能前，先用一句话起头 + 让 Claude 用 AskUserQuestion 工具反问你：技术实现、UI/UX、边界情况、取舍——逼出你没想到的硬骨头。
  • 谈清楚后，让它把结论写成 SPEC.md，然后开一个全新 session 拿着这份 spec 去实现（干净上下文 = 更好结果）。
  • 💡 这其实就是 /brainstorming 这类 skill skill 在做的事，但机制是官方的，没装 skill 也能直接让 Claude 面试你。

② 进入 Plan Mode（关键）官方

  • 用 Shift+Tab 切到 plan mode。Plan mode 下 Claude 只读不写，先探索代码 / 调研技术栈，产出一份实施计划。
  • 按 Ctrl+G 可以把计划在你的编辑器里打开直接改，再放行。
  • 何时跳过：能一句话描述清楚的小改动（改 typo、加一行 log、重命名）直接让它做，别为计划而计划。

③ 沉淀技术方案为文件官方

  • 让 Claude 把计划写成 PLAN.md / SPEC.md。好处：计划是文件 = 可审查、可版本化、可在新 session 里复用，不会随对话丢失。
  • 官方建议：最有用的 spec 是自包含的——点名涉及哪些文件与接口、写明什么不在范围内、并以一个端到端验证步骤收尾。
  • /writing-plans 这类 skill skill 可以把这步模板化。
💡 Tip：与其反复纠正 Claude 跑偏的输出，不如撤销改动、把计划改得更具体再重跑——官方原话「难的往往是想清楚该改什么」。一份你亲自审过的 plan，能让后面大半代码一次成型。 官方
2

搭好「上下文地基」（项目初始化）

目标：让 Claude 在整个项目周期里都「懂」你的代码和规范。

① /init 生成 CLAUDE.md官方

  • 在项目根目录运行 /init，它会扫描代码库（构建系统、测试框架、代码模式）生成 CLAUDE.md。
  • 写什么：Claude 猜不到的命令、与默认不同的代码风格、测试偏好、分支/PR 约定、项目特有架构决定、环境变量等坑。
  • 关键纪律：每行都问自己「删掉它 Claude 会不会犯错？」不会就删。CLAUDE.md 太长 = Claude 会忽略其中一半。关键规则可用 IMPORTANT / YOU MUST 加强。像对待代码一样定期精简它。

② 分层记忆官方

  • ~/.claude/CLAUDE.md（个人全局，所有项目生效）
  • ./CLAUDE.md（项目级，进 git，团队共享）
  • ./CLAUDE.local.md（个人项目私货，记得加进 .gitignore）
  • 父/子目录的 CLAUDE.md 会按需自动加载（monorepo 友好）；可用 @path/to/file 语法引入其他文件。

③ 偶尔才用的知识 → 放 Skill，别塞进 CLAUDE.md官方

  • CLAUDE.md 每次都加载，只放「广泛适用」的东西。只在特定场景才相关的领域知识/工作流，做成 .claude/skills/SKILL.md，Claude 按需加载，不污染每次对话。

④ 接入外部上下文与给富信息官方

  • MCP servers（claude mcp add）：连数据库、Figma、Sentry、Notion 等，让 Claude 直接读真实数据 / 设计稿。
  • 富信息输入：用 @ 引用文件、直接粘贴截图、给文档 URL、cat error.log | claude 管道喂数据。远比文字描述高效。

⑤ 配权限，减少打断官方

  • auto mode：分类器模型自动放行安全操作、只拦风险动作。
  • /permissions：把你信任的命令（如 npm run lint、git commit）加白名单。
  • /sandbox：OS 级隔离，限制文件/网络访问，让 Claude 在边界内更自由地跑。

Plan → Demo（快速原型）

目标：用最快速度做出能跑、能看、能点的原型，验证 idea 成不成立。

① 按计划分块实现AI 归纳

  • 让 Claude 按 PLAN.md 一个模块一个模块地实现，而不是「一次给我全部」。
  • /executing-plans、/subagent-driven-development 这类 skill skill 可以把计划拆成带 review checkpoint 的执行流。

② demo 阶段允许「糙」，但要能跑AI 归纳

  • 优先打通 happy path（核心主流程跑通），先不管边界情况。
  • 立刻运行并让它自己验证：让 Claude 跑 dev server、截图、和设计图对比、读报错，形成「写 → 跑 → 看 → 改」的闭环。/run、/verify 这类 skill skill 把这步包装好了。

③ 前端原型避免「AI 味」AI 归纳

  • /frontend-design 这类 skill skill 产出有设计感的界面；没装就直接把参考截图喂进去让它照着实现 官方。

④ 每个里程碑 commit官方

  • demo 能点通了就提交。加上 Claude 的自动检查点（/rewind），这是你的双重安全网。
💡 Tip：demo 阶段最忌「完美主义」。目标是最快拿到反馈（给自己、给用户、给合伙人看），而不是代码质量。质量留给阶段四。 AI 归纳
4

Demo → Product（工程化加固）

目标：把「能跑的 demo」变成「敢上线、能维护、不会半夜被叫起来」的产品——本质就是从 vibe coding 走向 agentic engineering。按重要性排序：

① 测试驱动 + 把验证「焊死」成闸门（最重要的护城河）官方

  • 让 Claude 先写会失败的测试（基于真实预期），你确认测试合理，再让它写实现直到测试通过、且不准改测试。/test-driven-development skill skill 把这流程固化。
  • 官方给了验证由松到紧的三档闸：prompt 内（同一句话要求跑检查并迭代，最轻）→ /goal 条件（设目标条件，独立评估器每轮复查，不达标继续）→ Stop hook（检查写成脚本挂 Stop 钩子，不通过不许结束回合，无人值守也能跑对）。
  • 让它给证据（测试输出 / 跑了什么命令 / 截图），别只说「好了」。

② 系统化调试，而不是「再试一次」官方

  • 描述清楚症状 + 可能位置 + 「修好」长什么样，并要求治根因、别压症状。/systematic-debugging skill skill 把复现→定位→改的流程模板化。
  • 善用回滚：改坏了 git reset --hard 或 /rewind 回到上一个好状态，比纠缠「修复」高效。

③ 对抗式代码审查（用新鲜上下文）官方

  • 最强一招：让一个 fresh subagent 只看 diff + 你给的标准去找 gap——它没参与写代码，不会偏袒。官方内建 /code-review skill 就是干这个的。
  • 或用 Writer / Reviewer 双 session：A 写、B 在干净上下文里审，把 B 的意见喂回 A。
  • /simplify（清冗余）、/security-review（安全审查）等 skill skill 按需补充。
  • ⚠️ 官方提醒：审查者被要求找问题就一定会报一些，盲目追每条会导致过度工程。让它只报影响正确性/需求的 gap，其余当可选。收到意见别照单全收（/receiving-code-review skill 讲的就是这个）。

④ 完成前强制验证官方

  • 声称「做完了 / 修好了」之前，必须真的跑命令、看输出、给证据。杜绝「我觉得应该好了」。/verification-before-completion skill skill 把它变成强制步骤。

⑤ 并行化扩展（项目变大后）官方

  • Git worktrees：多个任务在隔离的 git checkout 里并行，编辑互不冲突。
  • Agent teams：多 session 自动协调，带共享任务、消息、team lead。
  • claude -p fan-out：批量迁移/分析时，脚本循环调用非交互模式，配 --allowedTools 限定权限。

⑥ 自动化与防护栏官方

  • Hooks：在 .claude/settings.json 配 hook（如「每次编辑后跑 eslint」「禁止写 migrations 目录」）。和 CLAUDE.md 的「建议性」不同，hook 是确定性的、保证执行。让 Claude 帮你写，/hooks 浏览。
  • CI/CD：用 claude -p 接进 pre-commit / GitHub Actions，把测试/lint/构建自动化。

⑦ 收尾skill

  • /finishing-a-development-branch 这类 skill 帮你规范地 merge / 开 PR / 清理分支。

贯穿全程：上下文管理 + 提效杠杆

目标：既然上下文是根本约束，管好上下文就是最大的提效杠杆。 官方

上下文管理工具箱官方

  • /clear：不相关任务之间清空上下文。同一问题纠正超过两次，说明上下文已被失败尝试污染——/clear 重开 + 写个更好的 prompt，几乎总比硬聊下去强。
  • /compact <指令>：按需压缩并保留重点（如 /compact 重点保留 API 改动）。接近上限时也会自动压缩。
  • /rewind（Esc+Esc）：回到任意检查点，可只回对话 / 只回代码 / 都回。让你敢让 Claude「试点冒险的」，不行就倒回去。
  • /btw：问个不想留进上下文的小问题，答案浮层显示、不进历史。
  • subagent 做调研：研究代码会读一大堆文件、吃光上下文；丢给 subagent 在独立上下文里查、只回报摘要，主对话保持干净。

其它杠杆

  • 记忆：让它把约定写进 CLAUDE.md，跨 session 记住偏好。 官方
  • 自定义 skill / 命令：把反复做的流程封装成 .claude/skills/ 或 .claude/commands/——这个机制是官方的 官方，社区现成 skill（如 superpowers 那套）则是 skill。
  • 会话即分支：/rename 给会话命名、claude --continue / --resume 续上，每条工作线各自持久。 官方
  • Plan mode 当默认：养成「先 Shift+Tab 看计划再放行」的肌肉记忆。 官方

常见翻车模式（官方总结，照着避坑）官方

翻车症状解法
杂物间 session一个 session 里又干这个又问那个，上下文塞满无关信息不相关任务之间 /clear
反复纠正纠正了还错、再纠正还错，上下文全是失败尝试错两次就 /clear，带着教训重写 prompt
CLAUDE.md 过长规则太多，Claude 忽略其中一半狠心精简；能靠 hook 保证的就改成 hook
信任不验证看着对、实则漏边界永远给验证手段（测试/脚本/截图）；验证不了就别上线
无边界探索让它「调研一下」却不限范围，读几百个文件吃光上下文把调研范围收窄，或丢给 subagent

给你的 3 个最高 ROI 建议

1

永远先 plan 再 code，跑偏就改计划重跑——这一个习惯就能消除大半返工。 AI 归纳

2

给 Claude 一个能自己跑的验证手段，并把它当不可谈判项——官方说这是整篇最重要的一条，也是 demo 和真产品的分水岭。 官方

3

把重复流程沉淀成 CLAUDE.md + 自定义 skill——把这套方法迁移到任何新产品上即可。 AI 归纳
